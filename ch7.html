Software Life Cycle: Development phase
  1. Requirements analysis
  2. Design
  3. Implementation
  4. Testing
 the major steps in the traditional software development life cycle are requirements analysis, design, implementation, and testing
  1. specify what services the proposed system will provide, to identify any conditions (time constraints, security, and so on) on those services, and to define how the outside world will interact with the system
  2. creating a plan for the construction of the proposed system
  3. the actual writing of programs, creation of data files, and development of databases
  4. equated with the process of debugging programs and confirming that the final software product was compatible with the software requirements specification
Modular Implementation breaks down larger systems into basic building blocks to allow for more customization on demand
Unified Modeling Language (UML) general-purpose visual modeling language that is intended to provide a standard way to visualize the design of a system
Use Case describe how a system can be used to achieve specific goals or tasks
Use Case Diagram provides a visual representation of how users interact with a system
Data Relationships:
  1. one-to-one
  2. one-to-many
  3. many-to-many
  1. exemplified by the association between patients and occupied private rooms in that each patient is associated with only one room and each room is associated with only one patient
  2. exemplified by the association between physicians and patients in that one physician is associated with many patients and each patient is associated with one (primary) physician
  3. would occur if we included consulting physicians in the physician-patient relationship
Design Pattern predeveloped model for solving a recurring problem in software design
Software Quality Assurance (SQA) groups charged with overseeing and enforcing the quality control systems adopted by the organization
Software Testing:
  1. Glass box testing
  2. Black box testing
  3. Beta testing
  4. Alpha testing
  1. the software tester is aware of the interior structure of the software and uses this knowledge when designing the test
  2. tests that do not rely on knowledge of the software's interior composition
  3. a preliminary version of the software is given to a segment of the intended audience with the goal of leaning how the software performs in real-life situations before the final version of the product is solidified and released to the market
  4. similar to beta testing but performed at the developer's site
Documentation:
  1. User Documentation
  2. System Documentation
  3. Technical Documentation
  1. explains the features of the software and descrives how to use them
  2. describes the software's internal conposition so that the software can be maintained later in its life cycle
  3. describes how a software system should be installed and serviced
